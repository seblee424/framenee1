## 二、分阶段时间规划

### 第一阶段：后端 API 完善与部署（2~3 周）
> 目标：所有后端接口跑通，部署到阿里云函数计算

#### 第 1 周：阿里云 OSS 开通与对接
- 开通阿里云 OSS（对象存储）
- 创建 RAM 子账号（最小权限），获取 AccessKey
- 在 framene-server 中添加 OSS 上传接口（`POST /api/photos/upload`）
- 修改 photo_assets 表，url 指向 OSS 地址
- 测试：手机端上传图片到 OSS，Web 端能访问

#### 第 2 周：后端 API 全面对接
- 用户注册/登录 API 联调（手机验证码 + 邮箱 + 钉钉）
- 日历事件 CRUD API 联调（添加/修改/删除/查询）
- 日历成员/共享 API 联调
- Google 日历同步 API 完善（OAuth 回调 → 存储 token → 同步事件）
- 钉钉日历同步 API（参考 dingtalk-integration 文档）

#### 第 3 周：部署到阿里云函数计算 FC
- 将 framene-server 打包部署到阿里云函数计算（FC）
- 配置 FC 自定义域名（api.framene.com）
- 配置 FC 环境变量（RDS 连接串、OSS Key、DirectMail Key 等）
- 配置 HTTPS 证书（阿里云免费或购买）
- 压力测试，确保稳定运行

## 四、数据同步方案（核心架构）

```
┌──────────┐     ┌──────────────┐     ┌──────────┐
│ 手机端    │────▶│  阿里云 RDS  │◀────│  Web 端  │
│ (Flutter) │     │  (中心数据库) │     │ (React)  │
└──────────┘     └──────┬───────┘     └──────────┘
                        │
                  ┌─────▼──────┐
                  │ 阿里云 OSS  │
                  │  (照片存储) │
                  └────────────┘
```

**同步策略：**
- **日历事件**：以 RDS 为中心，写入后全设备可见
- **照片**：手机端拍照 → OSS 直传 → 记录 URL 到 RDS → Web/其他设备加载 OSS 图片
- **设备绑定**：生成设备码 → RDS 记录绑定关系 → 共享同一数据空间
- **实时性**：初期用轮询（每 30 秒），后期可加 WebSocket

## 五、上架方式对比

| 方式 | 优势 | 劣势 | 推荐度 |
|------|------|------|--------|
| **Google Play** | 全球覆盖，审核快，统一管理 | 国内用户需要翻墙 | ⭐⭐⭐⭐⭐ |
| **国内应用市场** | 国内用户直接下载 | 每个市场单独上架，材料繁琐 | ⭐⭐⭐ |
| **PWA（Web 端）** | 无审核，直接访问，可安装到桌面 | 功能受限（无推送通知） | ⭐⭐⭐⭐⭐ |
| **微信小程序** | 用户量大 | 审核严，功能受限，不适合相册类 | ⭐⭐ |

**推荐策略：**
1. Web 端以 PWA 形式上架 → 平板/笔记本直接访问（立即可用）
2. 安卓 App 上架 Google Play → 手机端
3. 国内用户通过 PWA 或 Google Play 下载（国内可以直连阿里云服务器，延迟低）

## 七、风险与注意事项

1. **阿里云 OSS 防盗链**：配置 Referer 白名单，只允许 framene.com 域名访问
2. **用户隐私**：照片涉及家庭隐私，务必:
   - 使用 HTTPS
   - 所有接口需 JWT 鉴权
   - OSS 文件设置为私有读，通过后端签名 URL 访问
3. **数据备份**：RDS 自动备份开启，定期手动备份
4. **成本控制**：OSS 设置 CDN 加速降低流量费，设置 OSS 生命周期自动清理临时文件
5. **国内网络**：函数计算 FC 在国内，国内用户访问延迟低；海外用户可通过阿里云全球加速
